# Session Journal Spec

> **Status:** v1, 2026-05-12. Sibling to [`journal-spec.md`](./journal-spec.md) which covers the task journal.
> **Owner:** HQ-core infra.

Two complementary journal patterns live side by side in HQ. Both write Markdown with YAML frontmatter; both are append-only during their active window.

| Pattern | Scope | Writer | Path | Purpose |
|---|---|---|---|---|
| **Task journal** (existing, since 2026-04) | One skill invocation (brainstorm, plan, deep-plan, prd, discover) | `.claude/skills/_shared/journal.sh` | `<project_dir>/journal/<ts>-<skill>-<short>.md` | Capture the deliberative trail for a *specific task* — decisions, open threads, findings, rejected approaches. Closed at handoff. |
| **Session journal** (new, this spec) | One Claude session, all activity | `core/scripts/session-journal.sh` + `/journal` command + hooks | `workspace/threads/journal/<YYYY-MM-DD>/<NNN>-<slug>.md` + `INDEX.md` | Capture working memory across the session so autocompact can throw away raw tool-result accumulation. Re-read on demand; never grows unbounded. |

This spec defines the **session journal**. The task-journal contract lives at [`journal-spec.md`](./journal-spec.md) (existing) and is unchanged.

## Why session journals exist

Pre-fix burn analysis (2026-05-10/11) showed cache_read tokens dominated cost: 80% of total token volume in long sessions came from re-reading the cached prefix on every turn. The prefix accumulated raw tool-results (Read outputs, Bash stdout, subagent returns) which can be lossily compressed into structured journal entries.

Cache stays **on**. Journal entries replace the *content* of the prefix, not the caching mechanism. A 200K-prefix session that re-reads 100 times costs 100 × 200K × 0.1 = 2M cache_read; the same session with a 30K-prefix re-read 100 times costs 0.3M cache_read — **6× reduction at the same caching discount.**

## File layout

```
workspace/threads/journal/
  2026-05-12/
    INDEX.md               # 1-line summary per entry, append-only
    000-session-start.md   # initial task + first reads
    001-pii-scan-spec.md   # an entry written mid-work
    002-test-fixtures.md   # next entry
    ...
  2026-05-13/
    INDEX.md
    ...
```

**Bucketing by date** (not by session UUID) is intentional: multiple Claude sessions in the same day share the directory and INDEX, so a continuation session (`/handoff` → new session) finds today's INDEX waiting without needing to look up a prior session ID. Cross-day continuity uses `/handoff`'s `handoff.json` plus the previous date's INDEX.

## Entry schema

```markdown
---
ts: 2026-05-12T16:45:00Z
session_short: a96880a8                  # first 8 chars of session UUID (optional)
title: PII scan deterministic pack       # human-readable
slug: pii-scan-spec                      # used in filename
files:                                   # files touched in this slice of work
  - scripts/promote-hq-core-scan.sh
  - scripts/tests/promote-hq-core-scan.test.sh
status: closed | active                  # active only for in-progress entries
---

# 001 — PII scan deterministic pack

## Goal
One-line statement of what this slice of work is about.

## Findings
Bullets. The interesting things learned, not the trivial things. Anything that would be lossy to compress further.

## Decisions
Numbered or bulleted. What was chosen, including the rejected alternatives if relevant.

## Next
What the next entry will pick up. Pointer to a story id, file, or open question.
```

**Size cap:** target ~50 lines of body, hard cap ~200 lines (≈5KB). Entries that need more space should be split (`002-pii-scan-spec-part-2.md`) or hand off to a full doc in `knowledge/`.

**INDEX.md schema:**

```markdown
# Session journal — 2026-05-12

- `000` 16:30Z — Session start: hq-token-economy WS-3 begin
- `001` 16:45Z — PII scan deterministic pack — scripts/promote-hq-core-scan.sh
- `002` 16:58Z — Test fixtures — 12 PII pattern assertions
- ...
```

One line per entry. Generated by the entry writer.

## Lifecycle

1. **Write** — `/journal "<title>"` or `core/scripts/session-journal.sh write "<title>"` writes a new entry. Numbering auto-increments from highest existing NNN. INDEX.md auto-updates.
2. **Auto-trigger** — `.claude/hooks/journal-due.sh` (PostToolUse) injects `JOURNAL ENTRY DUE` reminders after milestones (every 10 Edit/Write tool calls, successful `git commit`, successful test run, `/handoff` invocation). Counter tracked at `workspace/threads/journal/<date>/.tool-count`.
3. **Compact** — `.claude/hooks/journal-precompact.sh` (PreCompact) injects a reminder: "Before autocompact runs, write a journal entry summarizing the current prefix's findings/decisions. Post-compact, raw tool-results are gone but the journal entry remains."
4. **Continuation** — `.claude/hooks/load-journal-index-on-start.sh` (SessionStart) injects today's INDEX.md (and yesterday's if today's is empty) as a system reminder.
5. **Re-read** — `/journal --read <NNN>` or direct `Read` of the entry file. Cache makes the re-read 0.1× input rate.

## Interaction with existing systems

- **handoff.json** — extended (US-010) with an optional `journal_index_path` pointing at the most-recent INDEX.md so continuation sessions find it without scanning.
- **`/checkpoint`** — unchanged. Checkpoints capture session state; journal entries capture working memory. Different shapes, different lifetimes.
- **Task journals** (existing) — keep their behavior. When a brainstorm/plan/prd journal is active, that journal still gets the `decisions/findings/rejected` writes from its host skill. Session journal entries can reference task journals: `see {task-journal-relative-path}`.
- **Native session projects** — `.claude/hooks/auto-session-project.sh` covers non-trivial native Claude/Codex sessions that did not start with `/startwork`, `/brainstorm`, `/prd`, `/plan`, or `/run-project`. It calls `core/scripts/session-project.sh ensure`, which searches related project folders first and reuses a plausible match before creating a thin project with `prd.json`, `README.md`, `journal/`, and `sessions/`. The active project pointer lives at `.claude/state/active-session-project`. Native plan material from `ExitPlanMode` or Codex `update_plan` is archived through `.claude/hooks/native-plan-project-sync.sh` into that active project.
- **`/learn`** — unchanged. Learnings get promoted to policy files; journal entries are short-lived working memory.

```
INDEX.md
```

A future session compaction at 60% would keep only the active task and the last 2 turns; everything earlier in the prefix becomes "see `workspace/threads/journal/2026-05-12/INDEX.md`" — a single Read away when needed, cached at 0.1× thereafter.

## Cost model

| Scenario                                | Effective tokens per 100-turn session (avg 100K prefix) |
|-----------------------------------------|--------------------------------------------------------|
| No cache, no journal                    | 10M (100 × 100K × 1.0)                                |
| Cache on, prefix grows unbounded        | 1M (100 × 100K × 0.1)                                 |
| Cache on, journal-shrunk prefix (~20K)  | **200K (100 × 20K × 0.1)** + ~50K journal writes      |

Journal entries themselves cost output tokens to write (one entry ≈ 1K output ≈ 5K effective). Even 20 entries (100K effective) is cheap compared to 1M baseline cache_read.

## Non-goals

- **Replacing `/handoff`.** Handoff still owns the session-to-session boundary, just with a fatter "last journal" pointer now.
- **Replacing `/checkpoint`.** Checkpoints are full state snapshots; journals are append-only working memory.
- **Cross-machine sync.** Journal lives at `workspace/threads/`, which is local. If cross-machine continuation is needed, that's `hq-sync`'s problem, not this spec's.
- **Full structured search.** `qmd` over `workspace/threads/journal/` is fine if useful, but the primary access pattern is "read today's INDEX, pick the entry I need." No new index infrastructure.
