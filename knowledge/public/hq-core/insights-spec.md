# Insights Spec

## What is an Insight?

An **insight** is an educational or conceptual understanding extracted from a work session. Unlike policies (operational rules that prescribe behavior) or knowledge docs (domain reference material), insights capture *why things work the way they do* — patterns, mental models, and conceptual connections discovered during execution.

Insights are generated by the `Explanatory` output style (`★ Insight` blocks) and persisted via `/learn` with `source: "session-insight"`.

## How Insights Differ from Policies and Knowledge

| Dimension | Policy | Knowledge | Insight |
|-----------|--------|-----------|---------|
| Purpose | Prescribe behavior (NEVER/ALWAYS) | Document facts (what exists, how it works) | Explain understanding (why it works) |
| Tone | Imperative ("NEVER do X") | Descriptive ("X is configured at Y") | Educational ("X works because Y") |
| Enforcement | `hard` or `soft` | None | None |
| Location | `{scope}/policies/` | `knowledge/`, `companies/{co}/knowledge/` | `workspace/insights/` |
| Created by | `/learn` (rule routing) | Manual or `/handoff` step 0b | `/learn` (insight routing) via `/handoff` step 0c |
| Loaded before tasks | Yes (Standard Policy Loading Protocol) | On demand | On demand (via qmd search) |

## Directory Convention

Cross-cutting insights live in `workspace/insights/`. Company-scoped insights live inside the company's knowledge repo (respecting company isolation).

```
workspace/insights/
  INDEX.md                    # Directory map
  global/                     # Cross-cutting technical insights
    {slug}.md
  tools/                      # Tool/technology-specific insights
    {slug}.md
  concepts/                   # Pure conceptual/theoretical insights
    {slug}.md

companies/{co}/knowledge/insights/   # Company-scoped insights
  {slug}.md                          # Lives inside company knowledge git repo
```

## File Format

```markdown
---
type: insight
domain: [engineering]
tags: [topic-tags]
scope: global | company:{co} | repo:{repo} | tool:{tool}
source_session: T-{timestamp}-{slug}
created: YYYY-MM-DD
confidence: high | medium
relates_to: []
---

# {Title}

## Insight

Core conceptual understanding in 2-4 paragraphs. Educational, not directive.

## Context

When and why this matters. What situation makes this knowledge valuable.

## Example

Optional. Concrete example showing the insight in practice.
```

## Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Always `insight` |
| `domain` | list | From knowledge ontology: `engineering`, `product`, `operations`, `brand`, `market`, `growth`, `data` |
| `tags` | list | Free-form topic keywords for search |
| `scope` | string | `global`, `company:{co}`, `repo:{repo}`, or `tool:{tool}` |
| `source_session` | string | Thread ID from the session that generated this insight, or `"manual"` |
| `created` | date | ISO date of creation |
| `confidence` | enum | `high` (validated/proven) or `medium` (observed but not extensively tested) |

## Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `relates_to` | list | Paths to related insights, policies, or knowledge docs |
| `updated` | date | ISO date of last update (if merged/enriched) |

## Required Sections

- **Insight**: The core understanding (2-4 paragraphs)
- **Context**: When this matters

## Optional Sections

- **Example**: Concrete illustration
- **Related**: Links to policies, knowledge, or other insights

## Scope Subdirectory Mapping

| Scope value | Directory | Example |
|-------------|-----------|---------|
| `global` | `workspace/insights/global/` | `workspace/insights/global/hook-cant-see-conversation.md` |
| `company:{co}` | `companies/{co}/knowledge/insights/` | `companies/{company}/knowledge/insights/eventbridge-retry-reset.md` |
| `tool:{tool}` | `workspace/insights/tools/` | `workspace/insights/tools/qmd-bm25-vs-vector.md` |
| `repo:{repo}` | `workspace/insights/global/` | `workspace/insights/global/{product}-lambda-cold-starts.md` |
| Conceptual/theoretical | `workspace/insights/concepts/` | `workspace/insights/concepts/policy-vs-knowledge-vs-insight.md` |

**Company isolation**: Company-scoped insights live inside `companies/{co}/knowledge/insights/` — the company's own knowledge git repo. This ensures they are versioned, committed, and isolated alongside other company knowledge. `/learn` creates the `insights/` subdirectory if it doesn't exist.

## Slug Convention

- Kebab-case from title keywords, max 60 characters
- No scope prefix (subdirectory provides scope)
- Example: `eventbridge-retry-counter-resets-on-republish`

## Dedup Rules

Same qmd similarity thresholds as policies:

| Similarity | Action |
|------------|--------|
| > 0.85 | **Skip** — already captured |
| 0.6–0.85 | **Merge** — enrich existing insight, bump `updated` date |
| < 0.6 | **Create new** |

## Lifecycle

- **Creation**: Via `/learn` with `source: "session-insight"`, triggered by `/handoff` step 0c or `/checkpoint` step 0c
- **Discovery**: Via `qmd vsearch` or `qmd search` (indexed in `hq` collection automatically)
- **Maintenance**: `/garden` includes `workspace/insights/` in Scout scope; `/cleanup --consolidate-insights` deduplicates and prunes
- **Staleness**: Insights older than 90 days with `confidence: medium` and zero qmd search references are flagged as stale candidates by `/cleanup`
- **Promotion**: Insights that prove broadly valuable can be promoted to `knowledge/public/` as proper knowledge docs (manual, via `/garden`)

## Capture Rules

- Max 5 insights per `/handoff`, max 2 per `/checkpoint`
- Only persist genuinely reusable understanding, not session-specific facts
- Skip if session was purely mechanical (deploys, config, quick fixes)
- An insight about "why EventBridge retries reset on DLQ republish" = good
- "We deployed to staging" = not an insight, skip
- Insights never go in CLAUDE.md (no global promotion — discovery-only via qmd)
